## 1. 目标与定位

在井场监测平台上构建一个 **基于大模型的智能对话 Agent**，使运维人员能用自然语言完成环境数据查询、异常分析、知识问答等工作。

### 1.1 核心设计理念

- **Agent 是语义层**：所有数据查询、计算、分析均由已有后端服务（`sensor_service`、`alarm_service`、`analysis_service` 等）完成，Agent 通过 **Tool 调用** 获取结构化结果，再用大模型组织自然语言回答。Agent 不绕过已有服务直接操作数据库。
- **渐进式澄清**：Agent 在回答前先评估用户意图的清晰程度。若问题模糊或存在多种可能解读，Agent 应 **主动向用户追问** 以缩小范围；若问题已足够明确，则直接调用工具并回答，不做多余询问。
- **MCP + Tool + Skill 架构**：Agent 内部按 **MCP（Model Context Protocol）** 规范组织上下文与工具调用，Tool 层封装后端服务能力，Skill 层组合多个 Tool 完成复杂场景（如"环境异常诊断"是一个 Skill，内部编排 sensor + alarm + analysis 等多个 Tool 调用）。

### 1.2 分期策略

| 优先级 | 功能 | 说明 |
|--------|------|------|
| **P0** | 大模型直接对话 | 基础对话能力，接入 LLM，支持多轮会话 |
| **P0** | RAG 知识问答 | 基于文档/知识库的向量检索增强回答 |
| **P0** | 环境分析 Agent | 用户提问环境相关问题，Agent 自主查库分析并回答 |
| P1 | 报告生成 | 结构化日报、告警统计 |
| P1 | 告警智能分诊 | 关联上下文给出告警原因与建议 |
| P1 | 数据导出与推送 | 按需导出并推送到邮箱/企业微信 |
| P2 | 预测联动 | 调用预测模型回答趋势类问题 |
| P2 | 受控操作 | 带二次确认的设备控制编排 |

### 2.2 三层职责

| 层 | 职责 | 举例 |
|----|------|------|
| **MCP Context** | 管理会话上下文、系统提示词、可用工具声明、对话历史 | 根据当前模式（通用/RAG）装载不同的 system prompt 和 tool 列表 |
| **Tool** | 封装单个后端服务的原子操作，输入/输出均为 JSON Schema 声明 | `get_sensor_latest`、`get_environment_analysis`、`search_knowledge_base` |
| **Skill** | 编排多个 Tool 调用完成一个复杂意图，含前置条件判断、中间推理、结果整合 | "环境诊断"：先查传感器 → 查阈值 → 调分析 → 判断是否需要追问 → 组织回答 |

### 2.3 MCP 上下文管理

Agent 使用 MCP 协议规范管理模型交互的全生命周期：

- **System Prompt 模板化**：根据对话模式（通用/RAG）、已启用的 Tool/Skill 集合动态组装 system prompt
- **Tool 声明标准化**：每个 Tool 按 JSON Schema 声明参数与返回值，供 LLM function calling 消费
- **会话记忆**：维护多轮对话的 messages 列表，支持上下文窗口管理（截断策略、摘要压缩）
- **Tool 调用协议**：LLM 输出 tool_calls → Agent 执行 → 结果注入 messages → LLM 继续推理，循环直到产出最终回答

## 3. P0 功能：大模型直接对话与 RAG

### 3.1 大模型直接对话

用户通过自然语言与 Agent 交互，Agent 调用后端工具获取数据后回答。

**基础能力**：

- 接入大模型（OpenAI 兼容接口），支持 function calling / tool use
- 多轮对话：维护 session 级别的对话历史
- 流式输出：通过 SSE 逐段返回回答内容
- 工具调用透明化：Agent 在服务端完成工具调用循环，不向前端暴露中间 tool JSON

**对话模式**：

| 模式 | 说明 | 可用 Tool 集 |
|------|------|-------------|
| **通用对话** | 默认模式，可查数据、做分析、生成报告 | 全部已启用的 Tool |
| **知识问答（RAG）** | 优先走向量检索，回答附引用来源 | RAG Tool + 只读数据 Tool（用于交叉验证） |

模式可通过请求参数或会话级配置切换。

### 3.2 RAG 知识问答

对项目文档（`doc/`）、SOP、故障手册、设备手册等进行向量化，支持基于语义检索的增强问答。

**需求要点**：

- **文档入库**：支持 Markdown、PDF、TXT 等格式的文档分块（chunking）与向量化
- **向量存储**：首版可用轻量方案（如 ChromaDB、FAISS），演进可对接专用向量库
- **检索策略**：语义相似度检索 + 可选关键词混合检索（Hybrid Search）
- **引用溯源**：回答中必须标注来源文档及段落位置，减少无依据编造
- **知识更新**：支持增量更新，文档变更后可重新索引受影响的 chunk

**RAG 工具定义**：

| Tool 名称 | 功能 | 参数 |
|-----------|------|------|
| `search_knowledge_base` | 在向量库中检索相关文档片段 | `query`（检索语句）、`top_k`（返回条数，默认 5）、`filter`（可选，文档类型/来源过滤） |
| `get_document_content` | 获取指定文档的完整或分段内容 | `doc_id`、`chunk_id`（可选） |

### 3.3 接口设计

**对话接口**：

```
POST /api/agent/chat
```

请求：
```json
{
  "session_id": "optional-uuid",
  "messages": [
    { "role": "user", "content": "最近一小时温度怎么样？" }
  ],
  "mode": "general",
  "stream": true
}
```

非流式响应：
```json
{
  "content": "过去一小时内温度在 22.1℃ 到 28.4℃ 之间波动，平均 25.3℃，整体处于正常范围。",
  "session_id": "uuid",
  "clarification": null,
  "sources": [],
  "usage": { "prompt_tokens": 0, "completion_tokens": 0 }
}
```

流式响应（SSE）：
```
event: delta
data: {"content": "过去一小时内温度"}

event: delta
data: {"content": "在 22.1℃ 到 28.4℃ 之间波动"}

event: done
data: {"session_id": "uuid", "sources": []}
```

## 4. P0 功能：环境分析 Agent

### 4.1 功能描述

用户用自然语言提出环境相关问题，Agent 自主决策需要查询哪些数据、调用哪些分析接口，组织结构化分析结果为可读回答。

**典型场景**：

| 用户输入 | Agent 行为 |
|----------|-----------|
| "现在温度正常吗？" | 调 `get_sensor_latest` → 对比阈值 → 直接回答 |
| "过去一天环境有什么异常？" | 调 `get_environment_analysis(最近24h)` → 解读 anomalies → 回答 |
| "温度怎么样？" | **意图模糊**：追问"您想了解当前温度，还是某段时间内的温度趋势？" |
| "3 号设备最近一周湿度趋势如何？" | 明确意图，调 `get_environment_analysis(7d, device_id=3, bucket=1h)` → 归纳趋势 → 回答 |
| "为什么刚才温度告警了？" | 调 `get_alarms_history` + `get_sensor_history` + `get_environment_analysis` → 综合分诊 → 回答 |

### 4.2 渐进式意图澄清机制

这是本 Agent 的核心交互特性。Agent 不做两个极端——既不在信息充足时多余提问浪费用户时间，也不在信息模糊时猜测导致答非所问。

**澄清流程**：

```
用户提问
   │
   ▼
┌──────────────────┐
│ 意图解析 (LLM)    │
│ 识别：主题、时间、 │
│ 设备、分析类型     │
└────────┬─────────┘
         │
    ┌────┴────┐
    │ 足够明确？│
    └────┬────┘
     是 │    │ 否
        │    │
        ▼    ▼
   直接执行   生成澄清问题
   工具调用   （带选项建议）
        │         │
        ▼         ▼
   组织回答   等待用户回复
              → 再次评估
```

**澄清策略规则**：

1. **时间范围缺失**：用户未指定时间时，Agent 应追问或给出合理选项
   - 示例追问："您想看最近 1 小时、今天、还是最近一周的数据？"
2. **设备不明确**：存在多设备时，确认目标设备
   - 示例追问："系统中有 3 个监测设备（1 号井口、2 号储罐区、3 号管道），您想查看哪个？还是查看所有设备的汇总？"
3. **分析意图模糊**：区分"看当前值"与"看历史趋势"与"查异常"
   - 示例追问："您想了解当前的实时数据，还是一段时间内的趋势分析？"
4. **已足够明确**：当用户明确说出时间、设备、意图三要素中的关键项时，直接执行不追问
   - 示例："3 号设备过去 24 小时温度有没有异常" → 直接调工具回答

**澄清响应格式**：

```json
{
  "content": "您提到想了解温度情况，我想确认一下以便给您更准确的分析：",
  "clarification": {
    "question": "您想查看哪个时间范围的温度数据？",
    "options": [
      { "label": "最近 1 小时", "value": "1h" },
      { "label": "今天（00:00 至今）", "value": "today" },
      { "label": "最近 24 小时", "value": "24h" },
      { "label": "最近一周", "value": "7d" }
    ],
    "allow_custom": true
  },
  "session_id": "uuid"
}
```

前端收到 `clarification` 非空时，渲染选项按钮供用户快捷选择或自由输入。

### 4.3 环境分析 Skill 定义

**Skill: `env_diagnosis`（环境诊断）**

这是一个复合 Skill，编排多个 Tool 完成完整的环境分析流程：

```
env_diagnosis Skill 内部流程：
1. 解析用户意图 → 提取时间范围、设备、关注指标
2. [如信息不足] → 生成 clarification 返回前端
3. [信息充足] → 并行调用所需 Tool：
   - get_sensor_latest（如需实时值）
   - get_environment_analysis（如需统计/异常）
   - get_alarms_history（如涉及告警上下文）
4. 整合 Tool 返回的结构化数据
5. 由 LLM 组织为自然语言回答，包含关键数据点引用
```

### 4.4 Tool 清单（环境分析相关）

| Tool 名称 | 对应后端服务 | 功能 | 参数 |
|-----------|------------|------|------|
| `get_sensor_latest` | `SensorService.get_latest` | 获取最新传感器读数 | `device_id`（可选） |
| `get_sensor_history` | `SensorService.get_history` | 查历史采样数据 | `start_time`、`end_time`、`device_id`（可选）、`limit`（最大条数） |
| `get_environment_analysis` | `AnalysisService.get_summary` | 获取环境分析摘要（聚合、异常、趋势） | `start_time`、`end_time`、`device_id`（可选）、`bucket`（可选） |
| `get_alarms_history` | `AlarmService.get_history` | 查询历史告警记录 | `start_time`（可选）、`end_time`（可选）、`level`（可选）、`limit` |
| `get_alarm_config` | `AlarmService.get_config` | 获取当前告警阈值配置 | 无 |

**约束**：
- Agent 通过工具获取结构化数据后由 LLM 归纳解释，**禁止**在模型内自行计算 min/max/avg 等统计量
- 无数据时，Agent 如实告知"该时间段无数据"，**不编造数值**

## 5. P1 功能概述（后续展开）

以下功能 P1 优先级，当前仅描述需求轮廓，实现时再出详细需求。

### 5.1 报告生成

- 用户可用一句话触发结构化报告生成（日报、周报），输出 Markdown
- **定时日报**：每日固定时间（可在设置界面配置）自动生成当日告警统计与数据摘要
- **主动提醒**：Agent 定期轮询异常数据，若短时间内频繁出现异常，主动提醒用户查看
- 支持触发 CSV 导出（复用已有传感器/异常导出接口）

### 5.2 告警智能分诊

- 告警中心每条告警旁可提供"问 AI"入口
- Agent 自动拉取告警前后时间窗的传感器数据与环境分析，给出可能原因与建议动作
- 以 Skill 形式实现：`alarm_triage` Skill 编排 alarm + sensor + analysis 多个 Tool

### 5.3 数据导出与消息推送

- 按用户意图或定时策略导出环境数据与异常/告警信息
- 推送到指定渠道：邮箱（SMTP）、企业微信机器人
- 推送前可由模型生成摘要，原始数据以附件或下载链接为主

## 6. P2 功能概述（远期）

### 6.1 预测联动

- 调用已有 `temperature_forecast` 模型，回答"未来几小时会不会超限"
- 以 Tool 封装预测接口，Agent 可在环境诊断 Skill 中按需调用

### 6.2 受控操作

- 在严格二次确认下执行设备控制指令（机械臂预设动作、车辆行驶、模式切换等）
- Agent 只生成"动作计划"，前端弹确认框，用户确认后才下发
- 默认关闭（`AGENT_CONTROL_TOOLS_ENABLED=false`），按子能力细分开关
- 所有控制类操作写审计日志

## 7. 会话管理

### 7.1 会话存储

- **MVP**：`session_id` + 内存 LRU 字典保存最近 N 轮 messages；进程重启丢失可接受
- **演进**：Redis + TTL，与用户 ID 绑定

### 7.2 上下文窗口管理

- 对话历史超出模型上下文窗口时，采用截断 + 摘要压缩策略
- 保留最近 K 轮完整对话 + 更早轮次的摘要
- Tool 调用的原始返回如果过长，仅保留结构化摘要进入后续上下文

### 7.3 跨页面上下文携带

- 从环境监测/分析页跳转至 Agent 时，可携带当前时间范围、设备 ID、摘要片段作为隐藏上下文
- 减少用户重复描述当前场景

## 10. 安全与约束

- **工具隔离**：Tool 函数仅接收已校验参数，内部调用 Service，不执行任意 SQL/代码
- **限流**：`/api/agent/*` 按 IP/用户限速，防止 LLM 调用费用失控
- **敏感信息**：Tool 返回结果中不得包含 API Key、数据库密码等；日志中不打印完整提示词与模型返回
- **控制类工具**：默认关闭，开启后每次操作须前端二次确认 + 审计日志
- **RAG 安全**：文档入库前做基础清洗，向量检索结果不直接拼入 system prompt（走 user/tool message），降低 prompt injection 风险

---

## 11. 验收标准

### P0 验收

1. **基础对话**：用户提问，Agent 调用工具获取数据并回答，回答内容与工具返回数据一致
2. **多轮对话**：连续提问时 Agent 能理解上下文（如"那湿度呢？"承接前一轮的时间范围）
3. **流式输出**：SSE 连接正常，逐段返回文本
4. **RAG**：导入文档后，知识类提问能检索到相关片段并在回答中附引用
5. **意图澄清**：对模糊问题（如"温度怎么样"无时间范围），Agent 追问而非猜测
6. **明确问题直答**：对明确问题（如"3 号设备最近 1 小时温度最高多少"），直接调工具回答，不多余追问
7. **无数据诚实**：查无数据时明确告知，不编造数值
8. **模式切换**：通用模式与 RAG 模式的工具集和行为差异可感知

### 综合验收

9. Agent 工具仅调用已有服务，不自行拼接 SQL 或绕过服务层
10. 配置 `AGENT_ENABLED=false` 时所有 Agent 接口返回 503
11. 会话过期或清除后历史不残留
