# 后端需求 · 实时通信与 MQTT

## 1. WebSocket

### 1.1 端点

- 提供 WebSocket 端点（如 `/ws`），供 Web 前端建立长连接。

### 1.2 连接与保活

- 支持客户端发送 **心跳**（如 `{ "type": "ping" }`），服务端可响应 `pong` 或约定静默保活策略。
- 断线后由客户端重连；服务端应能安全处理重复连接与并发连接（MVP 可为广播式，不做用户级隔离）。

### 1.3 推送类型

至少支持以下 **业务消息类型**（`type` 字段区分），与 [`../Web端需求文档.md`](../Web端需求文档.md) 4.2 一致：

- `sensor_data`：环境采样；
- `vehicle_status`：车辆状态；
- `alarm`：告警事件。

消息体应包含统一的外层字段（如 `type`、`payload`、`timestamp`），便于前端分发。

## 2. MQTT

### 2.1 能力需求

- 支持通过环境变量配置启用/禁用 MQTT 客户端。
- 支持配置 Broker 地址、端口、认证信息、**订阅主题**（可多主题）。
- 接收传感器等业务载荷后，完成字段规范化、写入内部存储，并触发与 REST 一致的 WebSocket 广播。

### 2.2 身份与连接

- ClientId、网关/设备凭证等需可配置，避免与现场调试工具互踢；具体策略见实现与运维文档。

### 2.3 与模拟数据的关系

- 当关闭传感器模拟任务时，环境数据应主要来源于 MQTT（或后续其他接入），避免双源冲突。

## 3. 验收要点

- Web 端同一连接可收到三类推送；
- MQTT 上报一条后，REST `latest` 与 WS `sensor_data` 一致；
- 心跳不会导致连接被错误断开。
