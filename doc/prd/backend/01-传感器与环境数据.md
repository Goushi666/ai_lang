# 后端需求 · 传感器与环境数据

## 1. 功能需求

### 1.1 最新值查询

- 提供 `GET /api/sensors/latest`，返回当前最新一条传感器采样（温度、湿度、光照、设备标识、时间戳等）。
- **无数据时**：不得返回虚构占位数据；应返回可识别的「无数据」结果（如 HTTP 404 + 明确说明），供前端展示「--」或空状态。

### 1.2 历史查询

- 提供 `GET /api/sensors/history`，支持 `start_time`、`end_time`（ISO 8601）查询时间范围内的采样列表，按时间有序。
- 时间参数缺省或非法时，应返回明确错误（4xx），避免静默错误数据。

### 1.3 导出

- 提供 `GET /api/sensors/export`，在同一时间范围内导出 **CSV**，列至少包含：时间戳、设备 ID、温度、湿度、光照。

### 1.4 数据来源

- 支持从 **MQTT** 等外部通道接收 JSON 载荷，解析并写入与 REST/WebSocket 一致的内部模型（字段别名兼容应在实现层处理）。
- **默认不应**在服务端启动时预置大量模拟曲线数据；是否启用周期性模拟推送须可通过配置关闭，避免与真实设备数据混淆。

### 1.5 持久化与保留（与环境分析协同）

- MQTT（或其他接入）产生的 **有效采样须落库**（如 SQLite / 时序库），使 `GET /api/sensors/history`、导出与本项目 **环境分析** 读取 **同一数据源**；禁止仅写入与 HTTP 请求隔离的进程内副本而导致「历史为空、MQTT 有数」。
- **保留期限**：建议默认可配置（如 **7 天**），超期自动删除或等价不可查；具体变量名与行为见部署说明。
- **单实例与多实例**：多进程/多副本部署时，须约定 **单一共享存储** 或明确「每实例仅见本机数据」的限制（产品文档中写清）。

## 2. 数据语义

- **温度**：摄氏度（℃），浮点。
- **湿度**：相对湿度（%RH），浮点。
- **光照**：照度（Lux），浮点。
- **时间戳**：统一为 UTC 或可明确文档化的时区策略，与前端展示转换约定一致。

## 3. 与 WebSocket 的协同

- 每当产生新的有效传感器采样（含 MQTT 入库），应向 WebSocket 订阅方推送类型为 `sensor_data` 的消息，载荷字段与 REST 模型语义一致（见总体设计文档中的消息格式）。

## 4. 持久化与保留（支撑历史与环境分析）

以下条目为 **环境分析、长窗口统计** 的前置条件，与 [`08-环境分析.md`](./08-环境分析.md) §2 一致：

1. **MQTT（或其他接入）产生的有效采样须写入持久化存储**（如 SQLite，与 `SQLITE_URL` 等配置一致），`latest` / `history` / `export` 与 **环境分析** 读取 **同一数据源**；禁止 REST 与订阅侧各用独立内存库导致数据不一致。
2. **保留策略**：建议默认保留 **不少于 7 天**（可配置，如 `SENSOR_HISTORY_RETENTION_DAYS`）；超期记录 **删除或归档** 策略须在部署说明中写明。查询窗口超出已有数据范围时，返回空列表或明确空态，不虚构数据。
3. **设备维度**：每条记录带 `device_id`，与 MQTT 载荷规范化字段一致。

## 5. 验收要点

- 仅有真实上报时，latest/history/export 与推送内容一致；
- 无上报时 latest 不返回假数；
- 历史与导出在同一时间窗下记录条数与数值一致；
- 持久化开启时，MQTT 入库后 **同一进程/同一库实例** 下 history 可查到对应记录，且与环境分析摘要可对照一致。
