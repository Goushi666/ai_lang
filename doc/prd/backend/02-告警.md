# 后端需求 · 告警

## 1. 功能需求

### 1.1 历史告警查询

- 提供 `GET /api/alarms/history`，支持 `start_time`、`end_time` 查询时间范围内的告警记录。
- 单条告警至少包含：类型、级别、文案、触发值、阈值、是否已读、时间戳、唯一标识（若前端需标记已读）。

### 1.2 已读状态

- 提供 `PUT /api/alarms/{id}/read`（或等价路径），将指定告警标记为已读；返回操作是否成功。

### 1.3 告警阈值配置

- 提供 `GET /api/alarms/config` 获取当前阈值配置（如温度/湿度/光照上下限等，结构可扩展）。
- 提供 `PUT /api/alarms/config` 更新阈值；更新后新产生的告警判定应使用新阈值（MVP 可实现为内存配置）。

## 2. 实时推送

- 当系统判定产生新告警时，通过 WebSocket 推送类型为 `alarm`（或与 Web 端约定一致的类型名）的消息，载荷包含级别、文案、数值与阈值等，便于前端弹窗与列表追加。

## 3. 级别与类型

- 级别应支持多级（如低、中、高、紧急），与 Web 端展示颜色映射一致。
- 类型为字符串枚举或可扩展字符串，便于统计与筛选。

## 4. 验收要点

- REST 查询结果与推送产生的记录在字段上可对齐；
- 已读接口幂等或可重复调用不产生异常数据；
- 修改阈值后，模拟或真实数据触发规则符合新配置。
