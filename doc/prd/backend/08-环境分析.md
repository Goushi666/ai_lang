## 1. 术语与定位

### 1.1 什么是「环境分析」

在 **温度、湿度、光照** 等环境采样数据之上，针对用户选定的 **时间范围**（及可选 **设备**），提供：

- **统计聚合**：计数、最小值、最大值、平均值（可扩展分位数等）；
- **可选分桶序列**：如按小时，供趋势图展示；
- **异常识别**：按可配置规则判断是否存在越限、持续异常、数据不足等情况；
- **结构化输出**：结论码、提示列表、异常片段摘要，供 Web 与 **Agent 工具** 消费。

本模块 **不替代** `GET /api/sensors/latest` 与 `GET /api/sensors/history`；**不承担** MQTT 解析与写入逻辑（由传感器数据链路完成），分析层 **只读** 已持久化或可查询的采样数据。

### 1.2 边界说明

| 能力 | 归属 |
|------|------|
| 实时超阈推送、告警记录写入 | 告警模块（可与分析共用阈值配置） |
| 原始点列查询、CSV 全量导出 | 传感器 REST（`history` / `export`） |
| 自然语言长文解释、多轮问答 | Agent（通过工具拉取本模块结构化结果） |
| 统计、规则判定、异常列表 | **本模块** |

### 1.3 典型用户场景（端到端）

以下场景用于产品对齐与验收串联，**不改变**各模块边界：

1. **持续监测**：设备经 MQTT 上报温湿度、光照 → 后端写入 **与 `history` 查询同源** 的持久化存储（详见 [`01-传感器与环境数据.md`](./01-传感器与环境数据.md) 与保留策略）。
2. **回顾分析**：用户在 Web 端选择 **时间范围**（及可选 **设备**），请求环境分析摘要 → 展示 **min/max/avg、可选按小时趋势、结论码与短提示**。
3. **异常处置**：当存在规则识别的异常片段时，用户可在页面上查看列表，并 **导出异常 CSV**（与本模块导出接口对应），用于记录或交接。
4. **助手解读**（可选）：用户通过 Agent 用自然语言提问；Agent **仅通过工具** 拉取本模块结构化结果，**不**在模型内推算统计数据（详见 §6）。

## 2. 数据前置条件（与传感器需求协同）

以下条目与 [`01-传感器与环境数据.md`](./01-传感器与环境数据.md) 一致或为其延伸，**环境分析依赖其落地**：

1. **MQTT（或其他接入）收到的有效采样须写入持久化存储**（如 SQLite / 时序库），与 REST `history` 使用 **同一数据源**，避免「分析读到空、历史有数」不一致。
2. **时间戳**统一为 UTC（或与全局文档约定一致），分析请求中的 `start_time` / `end_time` 与库内比较规则一致。
3. **保留策略（推荐）**：环境采样保留 **不少于产品约定窗口**（如 **7 天**），超期自动删除或归档；具体天数可配置。分析接口在查询超出保留范围时，返回 **可理解的说明**（如无数据、部分无数据），不虚构数值。
4. **设备维度**：每条采样带 `device_id`；分析支持按设备过滤或「全设备」策略（首版需明确一种默认语义并在 OpenAPI 中写清）。

## 3. 功能需求

### 3.1 分析请求参数

- **必填**：`start_time`、`end_time`（ISO 8601），表示分析时间窗 `[start, end]`（闭区间或半开区间在接口文档中固定一种）。
- **可选**：`device_id`（不传时的语义：全局聚合或默认设备，须实现与文档一致）。
- **可选**：`bucket`  
  - `none` 或不传：不返回分桶序列，仅全窗聚合；  
  - `1h`：按 UTC 小时（或与全局约定时区）分桶，每桶内独立聚合。  
  非法取值返回 **400**。

### 3.2 时间窗统计聚合

在过滤后的采样集合上，对 **温度、湿度、光照** 分别计算：

- `count`：有效点数（缺失某字段的点是否计入由实现约定，须在 OpenAPI 说明）；
- `min`、`max`、`avg`（`avg` 为算术平均；`count=0` 时 min/max/avg 为空或 null）。

当时间窗内 **点数过多** 时，服务端允许 **均匀降采样** 后再聚合，响应中须包含 **`downsampled: true`** 及简要说明（或配置项上限与行为说明）。

### 3.3 分桶序列（可选）

- 当 `bucket=1h` 时，返回 `buckets` 数组；每个元素至少包含：**桶起始时间**、各指标的 **count/min/max/avg**（字段名与示例在 OpenAPI 固化）。
- 桶内无数据时该桶可省略或显式标记为空，与前端约定一致。

### 3.4 异常识别（规则型，首版）

在 **同一时间窗、同一过滤条件** 下，除聚合外输出 **异常相关结构化信息**。首版采用 **可配置参数 + 规则引擎**（非机器学习），规则示例（可实现子集，但须在版本中列出已支持项）：

| 类型（示例） | 含义（产品描述） | 规则方向（供实现） |
|--------------|------------------|-------------------|
| 数据不足 | 窗口内有效点数低于下限 | `count < N_min` |
| 温度持续偏高 | 温度连续 k 个点高于阈值 | 与告警 `temperature_threshold` 或单独「分析用上界」联动 |
| 湿度/光照越限 | 单次或持续超阈 | 与告警配置或独立舒适区配置 |
| 稳定/正常 | 各指标在舒适区内且无持续越限 | 默认结论之一 |

**优先级 / 结论码**：当多条规则同时满足时，须有 **确定性优先级**（如数据不足优先于其他），并映射到 **`summary_code`**（字符串枚举，如 `insufficient_data`、`temp_spike`、`humidity_high`、`stable`、`mixed` 等，具体列表首版冻结一版）。

**异常片段**：对「持续类」异常，须同时在 **摘要 JSON** 与 **导出 CSV** 中可对应，字段语义一致。

- 摘要响应中建议使用字段名 **`anomalies`**：`array`，无异常时为空数组。
- 每项对象建议字段（与 §3.6 导出列对齐）：`device_id`、`metric`（`temperature` | `humidity` | `light`）、`type` 或 `rule_id`（规则/异常类型标识）、`start_time`、`end_time`（ISO 8601）、`peak`（可选，窗口内代表值或峰值）、`threshold`（可选，判定所用阈值）。

便于前端表格展示、时间轴高亮与 CSV 导出同源。

### 3.5 结构化摘要与提示

- **`summary_code`**：单一主结论码，供 UI 图标与 Agent 快速理解。
- **`summary_hints`**：若干条 **短中文（或 i18n key）**，描述可读结论，**不替代** Agent 生成的长文。
- **`framework` 或等价字段**：实现过渡期可标记「占位」；正式版本应为 false 或废弃该字段（与前端约定）。

### 3.6 异常信息导出

- 提供 **独立导出能力**（推荐路径示例：`GET /api/analysis/environment/anomalies/export`），与「全量传感器 CSV」区分：
  - **查询参数**：与摘要接口一致的 `start_time`、`end_time`、`device_id`（可选）；可额外支持按 `summary_code` 或规则类型筛选（演进）。
  - **格式**：首版 **CSV** 即可；列至少包含：异常类型/规则标识、指标名、开始时间、结束时间、峰值或代表值、阈值、设备 ID。
  - **空结果**：无异常时返回 **空表头 + 0 行** 或仅表头，与 `01` 导出语义一致，**200** 响应。
  - **行数上限**：超大窗口可限制最大导出行数并返回提示或 413/400（与 OpenAPI 一致）。

### 3.7 触发式分析（与查询等价）

- **`POST /api/analysis/environment/run`**（或等价路径）：请求参数与 **GET 摘要** 一致，执行 **同一次计算** 并返回相同结构；用于前端「刷新分析」按钮。
- **不要求**首版实现异步任务队列；若未来引入长任务，再增加任务 ID 与轮询/SSE（见演进）。

### 3.8 与告警模块的关系

- 分析服务 **可读** 告警阈值配置（[`02-告警.md`](./02-告警.md)），用于越限与舒适区判断；**阈值展示**与 **分析结论** 中引用阈值时，须与配置一致。
- **自动产生新告警**：首版默认为 **关闭**；若开启须 **独立配置开关**，避免与实时 MQTT 侧告警 **重复风暴**。

## 4. 接口需求（REST）

**路径前缀**：`/api/analysis`（已在实现中采用，OpenAPI 以此为准）。

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/environment/summary` | 见 §3；返回聚合、buckets、summary_code、summary_hints、异常片段、downsampled 等 |
| POST | `/environment/run` | 与 GET 摘要等价的一次计算 |
| GET | `/environment/anomalies/export` | 导出异常片段 CSV（§3.6） |

**错误与空数据**：

- 时间非法、顺序颠倒、`start > end`：**400**。
- 窗口内无采样：建议 **200** + `count=0`、明确 `summary_code`（如 `insufficient_data`）及 hints；若产品选择 **404**，须全链路（前端、Agent 工具）统一约定。

**鉴权**：与 [`06-安全与配置.md`](./06-安全与配置.md) 一致；MVP 可为可选鉴权，演进后分析结果可能含敏感运营数据，需与全局策略对齐。

## 5. WebSocket（可选 / 演进）

- 若后续支持 **定时分析** 或 **长任务完成** 通知，可增加消息类型 `analysis_summary`，载荷仅含 **精简字段**（如时间窗、`summary_code`、生成时间），详情由客户端再调 REST。
- **首版可仅 REST**，不在本版本强制实现。

## 6. 与智能 Agent 的协作

与 [`09-智能Agent.md`](./09-智能Agent.md) 一致：

- 提供工具 **`get_environment_analysis`**（名称以实现为准），参数映射 **`start_time`、`end_time`、`device_id`、`bucket`**，返回值为 **本模块 JSON**（可序列化子集）。
- **禁止**在模型侧对原始时序做精确 min/max/avg 或异常判定；**一律**由本模块计算后供模型 **归纳、解释、对比多时段**。
- Agent **不参与**写传感器数据、不替代告警联锁；**不参与**环境「控制决策」，仅辅助 **查询与说明**。

## 7. 非功能需求

- **性能**：单次分析在常规数据量下 **P95 完成时间** 目标可参照 [`07-非功能与演进.md`](./07-非功能与演进.md)（如 3 秒内）；超时返回 **504** 或降级策略（仅全窗聚合、不返回分桶）须在文档中说明。
- **可配置**：`ANALYSIS_ENABLED`、最大点数/降采样、`N_min`、连续点数 k、保留天数（若在分析侧校验）等通过环境变量或配置中心管理，**禁止**硬编码密钥。
- **可观测性**：关键错误打日志；不在日志中输出完整原始采样大报文（可采样或截断）。

## 8. 配置项（建议清单）

| 配置项 | 说明 |
|--------|------|
| `ANALYSIS_ENABLED` | 关闭时摘要接口可返回 503 或明确错误 |
| `ANALYSIS_MAX_POINTS` | 超过则降采样后再聚合 |
| `ANALYSIS_MIN_POINTS` | 低于则 `insufficient_data` |
| `SENSOR_HISTORY_RETENTION_DAYS` | 与持久化模块协同，默认如 7（见 §2） |

（连续越限点数 k、是否与告警阈值共用等可在实现阶段落入配置或数据库，本文档要求 **行为可配置或可文档化**。）

## 9. 验收标准（抽检）

1. MQTT 入库后，在保留期内任选时间窗调用 **`/environment/summary`**，聚合结果与同一窗口 **`/api/sensors/history`** 手工核对一致（允许降采样时标注 `downsampled`）。
2. 构造超阈/连续高点数据，`summary_code` 与 **异常片段** 符合规则定义；**导出 CSV** 列完整且与片段一致。
3. `device_id` 过滤后仅统计该设备数据。
4. `bucket=1h` 时桶边界与文档约定一致（与前端联调确认）。
5. Agent 工具仅返回服务端计算结果，**无数据时不编造数值**（与 Agent 验收联合进行）。
