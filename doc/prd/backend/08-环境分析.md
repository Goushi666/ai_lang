# 后端需求 · 环境分析

## 1. 目标

在已有传感器 **历史与实时数据** 基础上，提供 **统计聚合、趋势摘要、异常片段识别** 等能力，供 Web 端展示与智能 Agent 作为工具调用；**不替代** 硬件采样与原始 `history/latest` 接口。

## 2. 功能需求

### 2.1 时间窗聚合

- 支持指定 `device_id`（可选，缺省为全局或默认设备）、`start_time`、`end_time`，返回该窗口内：
  - 温度 / 湿度 / 光照的 **计数、均值、最小、最大**；
  - 可选 **按固定桶**（如按小时）的序列，用于前端柱状或阶梯图。
- 时间范围过大时，后端可限制最大跨度或降采样，并返回明确提示或分页策略。

### 2.2 分析摘要（结构化）

- 提供 **单次请求** 生成的 **结构化摘要** JSON，例如：
  - 各指标是否在常见舒适区间内（规则可配置或与告警阈值联动）；
  - 是否存在 **持续偏高/偏低** 时段（简单规则：连续 N 个点超阈或滑动均值偏离）；
  - 可选 **简短结论代码**（如 `stable` / `temp_spike` / `insufficient_data`），供 UI 图标映射；**自然语言长文** 可由 Agent 层生成，本模块以结构化字段为主。

### 2.3 触发式分析任务（可选演进）

- 支持 **按需触发** 一次完整分析（与 2.1+2.2 合并的一次计算），便于「刷新分析」按钮；不要求首版必须异步队列。
- 若后续引入 **定时分析**，应可配置 Cron/间隔，结果写入缓存或表，并通过 WebSocket 推送（见 `05` 与下文）。

### 2.4 与告警模块的关系

- 分析服务 **可读** 告警阈值配置，用于区间判断与文案中的阈值引用。
- **是否自动产生新告警**：首版可为「仅分析展示」；若开启联动，须单独配置开关，避免与现有实时阈值告警重复风暴。

## 3. 接口需求（REST）

路径前缀建议：`/api/analysis`（或 `/api/sensors/analytics`，二选一在 OpenAPI 固化）。

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/summary` 或 `/environment/summary` | 查询参数：`start_time`、`end_time`、可选 `device_id`、可选 `bucket`（如 `1h`）；返回聚合 + 结构化摘要 |
| POST | `/environment/run`（可选） | 触发一次与 GET 等价的重算并返回；用于显式「刷新」 |

错误语义：

- 时间参数非法：400；
- 窗口内无数据：200 + 空数据标记或 404，需在 OpenAPI 与前端约定一致。

## 4. WebSocket（可选）

- 若存在 **定时分析** 或 **长任务完成** 推送，可增加消息类型如 `analysis_summary`，载荷为最近一次分析结果 ID 或精简摘要；**首版可仅 REST**。

## 5. 非功能需求

- 单次分析应在可接受时间内完成（如 P95 &lt; 3s，数据量受限于仓库或 DB）；超时可返回 504 或降级为粗粒度聚合。
- 实现须 **与 Agent 工具层解耦**：Agent 仅调用本模块公开 Service/API，不在 LLM 内做数值统计。

## 6. 数据依赖

- 依赖 **SensorRepository**（或持久化后的等价查询）；内存条数上限下，分析窗口与产品预期一致即可。
- 长期需 **持久化/时序库** 才能支撑周级、月级分析（见 `07-非功能与演进.md`）。
